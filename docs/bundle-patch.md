# Bundle patch — `meshtrove.bundle-patch/1`

The interchange format a **metadata scraper** produces and MeshTrove consumes:
a *partial* description of a bundle — real names, tags, sample images, a
description — merged onto a bundle that already exists in the library. The
library side knows about files, variants and renders; the patch side knows what
the shop page said. The patch never creates or deletes models: it decorates
members the bundle already has.

Ground truth: the consumer is `backend/src/routes/patch.rs`. If this document
and the code disagree, the code wins — then fix this document.

## The deliverable

One zip:

```
<anything>-patch.zip
  patch.json          the document (exactly one; root or any folder)
  images/gold.png     every other file entry is an image, keyed by its
  images/_cover.jpg   full path inside the zip
```

- `patch.json` is found by name at any depth (`patch.json` or `*/patch.json`).
  Ship exactly one.
- **Every** other file entry is treated as an image. The `image` /
  `bundle.images` fields reference them by their full zip path
  (`"images/gold.png"`), and the reference must match the entry name
  **byte for byte** — no URL-encoding, no entity-encoding, no leading `./`.
- The zip is buffered in memory server-side. Ship sample renders and covers,
  not print files; tens of megabytes is fine, gigabytes is not.

## The document

Everything is optional. **An absent field says nothing and changes nothing** —
that is the core rule, and it is what lets a scraper that knows only names and
tags merge over a library that knows everything else. Unknown fields are
ignored (a richer patch still applies), so you may carry extra provenance for
your own use.

```jsonc
{
  "schema": "meshtrove.bundle-patch/1",   // REQUIRED by convention (see Versioning)
  "source": {
    "site": "lootstudios",                // provenance; not read
    "url": "https://…/bundle/buried-tomb/", // READ: becomes source_url (see merge rules)
    "scraped_at": "2026-07-14T20:00:00Z"  // provenance; not read
  },
  "bundle": {
    "name": "Buried Tomb",                // READ: real title (replaces autogenerated names only)
    "creator": "Loot Studios",            // READ: resolved/created by name
    "description_md": "**tagline**\n\nbody…", // READ: markdown; becomes a new revision
    "images": ["images/_cover.jpg", "images/_hero1.jpg"], // READ: covers, primary FIRST
    "categories": ["Heroes", "Enemies"],  // not read (yet)
    "extras": [ { "label": "Magazine", "url": "https://…" } ] // not read (yet)
  },
  "models": [
    {
      "uuid": "ef23a690-…",               // READ: the library model's id, when known —
                                          // strongest match signal ("" reads as absent)
      "match": {                          // how to find this model in the library
        "name": "Gold",                   // the site's name for it
        "aliases": ["Gold", "gold-32mm"]  // other spellings it might be filed under
      },
      "name": "Gold",                     // rename target (applied only if the user opts in)
      "tags": ["catfolk", "bard", "heroes"], // model tags (what it IS) — lowercase them
      "description_md": "…",              // READ: markdown; becomes a model description revision
      "image": "images/fn2110ac01.png",   // ONE sample render, zip path, or null
      "category": "Heroes",               // the shop category; disambiguates duplicates
      "source_ref": "FN2110AC01"          // the site's own id; not read (yet)
    }
  ]
}
```

Fields marked *not read* are legal and encouraged provenance; the merger
ignores them today and may use them later.

### Text encoding

All read text fields (`source.url`, `bundle.name/creator/description_md`,
`models[].name/category/tags/description_md`, `match.name/aliases`) are
**HTML-entity-decoded on ingest** — `&amp;` → `&`, `&#8211;` → `–` — so a scraper that lifts strings
straight out of markup doesn't have to clean them. The two exceptions are
`bundle.images` and `models[].image`: image references are left untouched so
they keep matching zip entry names byte for byte. Prefer emitting clean text
anyway; the decoding is a safety net, not a feature to lean on.

## Matching: how a patch model finds its library model

Resolution is three tiers, per patch model. Tier 0 is by id; the name tiers
run against every bundle member's name **and its stored aliases** (names
taught by earlier patches/renames):

0. **By uuid.** A patch model carrying `uuid` points at the member itself —
   the strongest signal, and it decides alone: names never overrule it. A
   uuid that is *not* a member of this bundle matches **nothing** (no name
   fallback — the id says the scraper meant one specific model, and it isn't
   here); the row surfaces as unmatched for hand assignment. `""` and `null`
   mean "no uuid"; any other non-uuid string rejects the whole patch.
1. **Exact, collapsed.** Every spelling the patch offers — `name`,
   `match.name`, each of `match.aliases` — is collapsed (strip every
   non-alphanumeric character, lowercase) and looked up. `"Warrior Mummy"`
   matches the `WarriorMummy` folder-model outright.
2. **Prefix fallback** (only when tier 1 finds nothing). Scraped names often
   trail the bundle or category ("Crocodile *Buried Tomb*"): each side is
   split into words (camelCase expanded first), and a member matches when one
   of its spellings is the *leading run* of one of the patch's. Ties go to the
   longest member name (`WarriorMummy` beats `Warrior`), then to members whose
   tags carry the patch model's `category`.

The result per patch model is zero, one, or several candidates. One candidate
auto-applies — **unless contested** (two patch entries whose sole match is the
same member, e.g. a "Gold" in Heroes and a "Gold" in Busts landing on one
"Gold" model): contested and ambiguous rows never auto-apply; the preview
surfaces them and the user assigns each by hand. Unmatched patch models do
nothing (they are *not* created); unmatched members are left untouched.

**Scraper guidance:**

- If you know the library model's id (from the API, an export, or a
  hand-matched list), emit it as `uuid` — matching is then exact and the
  review has nothing to question.
- Failing that, if you know the library's exact *name* for a model, put it in
  `match.aliases` — an exact collapsed match ends the search before any
  guessing starts.
- Emit the spellings a carve is likely to have produced: the squashed name
  (`WarriorMummy`), a slug (`warrior-mummy`), an underscored slug. Harmless
  when redundant, decisive when the folder name was the model name.
- Always set `category` when the site has one — it is the tiebreak between
  same-named models, and it reads well in the review UI.
- A patch model with no name and no match hints can never match; don't emit it.

## Merge semantics: what apply does, field by field

Merging is two calls, mirroring every other MeshTrove write: **preview**
returns the match diff for confirmation, **apply** writes it. Nothing is
written until apply, and apply only touches what the options ask for.

| Patch field | On apply |
|---|---|
| `bundle.name` | Replaces the bundle's name **only if the current name is autogenerated** (came from an archive filename, never typed by a user). Slug regenerated, keeping its token so old URLs redirect. The autogenerated flag stays set — a later, better scrape may correct it again. |
| `bundle.creator` | Resolved to a `creators` row case-insensitively by name, created if missing. **Replaces** the bundle's creator. **Fills** each matched model's creator *only where the model has none*. |
| `source.url` | **Replaces** the bundle's `source_url` (the scrape *is* the bundle's page). **Fills** each matched model's `source_url` only where empty. |
| `bundle.description_md` | Inserted as a new description **revision** (newest = current; history kept). Gated by `bundle_description`. Blank/whitespace = absent. |
| `bundle.images` | Gated by `bundle_cover`. All added to the bundle; the **first becomes primary**, existing primaries demoted (not deleted). Deduped by content hash — re-applying the same patch adds nothing. |
| `models[].name` | A **per-model, opt-in rename** (`options.rename`); never automatic. Slug regenerated with its token kept. The old name is recorded as an alias. A *declined* rename is also recorded as an alias, so the next scrape recognises the model and doesn't re-offer it. |
| `models[].tags` | **Model** tags (what the model *is*) — never variant tags. Upserted into the shared tag vocabulary. Mode `merge` (default: add, never remove), `replace` (drop the model's tags first), or `skip`. |
| `models[].description_md` | Inserted as a new **model** description revision (newest = current; history kept). Gated by `model_descriptions`. Skipped when it equals the model's current description, so re-applying a patch doesn't stack duplicate revisions. |
| `models[].image` | One image per model, made **primary**. Mode `replace_generated` (default: delete the model's auto-rendered previews so the scrape's shot is what shows), `add` (keep renders alongside), or `skip`. Deduped by content hash. |
| `match.aliases` | Every alias is stored on the matched model (deduped case-insensitively, ones equal to the model's final name dropped) — the patch *teaches* the library its other names, so the next import or patch resolves exactly. |

Everything runs in one transaction; image blobs enter the content-addressed
store before it opens, so a failed apply leaves no half-merged bundle.

## API

Both endpoints require edit rights on the bundle and take
`multipart/form-data`:

```
POST /api/bundles/{id}/patch/preview
  file:    the patch zip
→ 200 {
    bundle_description,        // what the patch carries, or null
    bundle_covers: [dataUrl],  // primary first
    matched:        [{key, patch_name, model_id, model_name, add_tags, has_image, has_description, category}],
    ambiguous:      [{key, patch_name, patch_tags, has_image, has_description, category, candidates}],
    unmatched_patch:[{…, candidates: []}],   // user may still assign manually
    unmatched_members: [name],               // members the patch says nothing about
    members:        [{id, name, tags, aliases}]  // full list, for manual assignment
  }

POST /api/bundles/{id}/patch
  file:    the same zip
  options: JSON text field {
    rename:        ["3", "7"],          // patch-model KEYS whose rename to apply
    model_tags:    "merge" | "replace" | "skip",           // default merge
    model_images:  "replace_generated" | "add" | "skip",   // default replace_generated
    model_descriptions: bool,           // default false
    bundle_cover:       bool,           // default false
    bundle_description: bool,           // default false
    matches: { "5": "<member-uuid>" }   // resolve ambiguous/contested/unmatched rows,
  }                                     // or override an automatic match
→ 200 { models_updated, images_added, tags_added, aliases_added, descriptions_added }
```

`key` is the patch model's **index** in `models[]` — its stable identity across
preview and apply. Names are not unique (two "Gold"s), so everything keys on
the index; keep `models[]` in the same order in both calls (send the same zip).

## Versioning

`schema` must be `"meshtrove.bundle-patch/1"`. The current consumer does not
reject other values, but do not rely on that: emit it, and treat any future
`/2` as a different format. Within `/1`, evolution is additive — new optional
fields may appear, absent still means "no change", and consumers ignore what
they don't know.

## A minimal, valid patch

Names and tags only — no images, nothing bundle-level:

```json
{
  "schema": "meshtrove.bundle-patch/1",
  "source": { "url": "https://example.shop/my-bundle" },
  "models": [
    { "match": { "name": "Warrior Mummy" }, "tags": ["undead", "medium"] }
  ]
}
```

Zip it with just that `patch.json` inside and it is a complete deliverable.
